# 第一章 系统概述

## 1.1 系统简介

RecSync（Recording Synchronization）是一款专业的多设备同步视频录制系统，专为运动科学研究、体育训练分析和生物力学实验设计。系统采用分布式架构，支持多台设备同时录制，并通过软件同步技术实现高精度的时间同步。

### 1.1.1 设计目标

RecSync系统的设计目标是为运动科学研究提供一个低成本、高精度、易于使用的多机位同步录制解决方案。传统的多机位同步录制系统通常需要昂贵的硬件同步设备，而RecSync通过软件同步技术，大幅降低了系统成本，同时保持了足够的同步精度。

### 1.1.2 核心理念

系统采用Leader-Client架构，一台计算机作为Leader（控制端），负责协调和控制多台Client（录制端）设备。所有Client设备通过局域网连接到Leader，接收统一的录制指令，实现同步录制。

### 1.1.3 技术创新

RecSync系统的核心技术创新包括：
- 基于UDP的低延迟RPC通信机制
- 简化的网络时间协议（SNTP）实现时钟同步
- 自动服务发现机制（mDNS + UDP广播）
- 分块文件传输协议，支持断点续传和MD5校验
- 实验数据管理系统，支持受试者、动作、回合的结构化管理

## 1.2 主要功能

### 1.2.1 多设备同步录制

系统支持最多10台Client设备同时录制，所有设备接收统一的开始/停止指令，实现毫秒级的同步精度。每台Client设备可以独立配置摄像头参数，包括分辨率、帧率等。

### 1.2.2 自动服务发现

Client设备启动后，可以自动发现局域网内的Leader设备，无需手动配置IP地址。系统同时支持mDNS和UDP广播两种发现机制，确保在各种网络环境下都能正常工作。

### 1.2.3 实时客户端监控

Leader端实时显示所有已连接的Client设备状态，包括设备名称、IP地址、连接状态、同步状态等。管理员可以随时查看系统整体状态。

### 1.2.4 实验数据管理

系统提供完整的实验数据管理功能，支持：
- 受试者ID管理（如s01, s02...）
- 动作ID管理（如m01, m02...）
- 回合自动递增（如e1, e2, e3...）
- 重测功能（如e1_retake1, e1_retake2...）
- 批次管理，每次录制会话生成唯一的批次ID

### 1.2.5 视频参数配置

支持灵活的视频参数配置：
- 分辨率：640x480、1280x720、1920x1080
- 帧率：15fps、24fps、30fps、60fps
- 所有Client设备统一使用Leader端配置的参数

### 1.2.6 文件自动传输

录制完成后，Client设备自动将视频文件上传到Leader端的归档目录。文件传输采用分块传输协议，支持进度显示和MD5校验，确保文件完整性。

### 1.2.7 文件命名规范

系统采用结构化的文件命名规范：
- 正常录制：`{设备名}_{受试者ID}_{动作ID}_{时间戳}_{回合ID}.mp4`
- 重测录制：`{设备名}_{受试者ID}_{动作ID}_{时间戳}_{回合ID}_retake{N}.mp4`

例如：`front_s01_m01_20240115143022_e1.mp4`

## 1.3 应用场景

### 1.3.1 运动生物力学研究

在运动生物力学研究中，研究人员需要从多个角度同时记录运动员的动作，以便进行三维运动分析。RecSync系统可以提供多机位同步录制，满足研究需求。

### 1.3.2 体育训练分析

教练可以使用RecSync系统记录运动员的训练过程，从不同角度观察技术动作，发现问题并进行针对性指导。系统的实验数据管理功能可以帮助教练系统化地管理训练数据。

### 1.3.3 康复医学评估

在康复医学领域，医生需要记录患者的运动功能，评估康复效果。RecSync系统可以提供标准化的录制流程，便于前后对比分析。

### 1.3.4 动作捕捉预处理

对于需要进行动作捕捉的项目，RecSync系统可以作为预处理工具，提供多角度的视频参考，辅助标记点识别和轨迹重建。

## 1.4 系统特点

### 1.4.1 低成本

相比传统的硬件同步系统，RecSync采用纯软件同步方案，无需购买昂贵的同步设备。只需要普通的计算机和摄像头即可搭建完整的多机位录制系统。

### 1.4.2 易部署

系统采用便携式设计，无需安装，解压即用。支持Windows平台，兼容性好。网络配置简单，支持自动发现，降低了部署难度。

### 1.4.3 高精度

虽然采用软件同步，但系统通过优化的时间同步算法和低延迟通信机制，可以实现毫秒级的同步精度，满足大多数应用场景的需求。

### 1.4.4 可扩展

系统采用模块化设计，核心功能封装在recsync-core模块中，可以方便地集成到其他应用中。支持自定义RPC消息，可以扩展新功能。

### 1.4.5 易使用

系统提供友好的图形界面，操作简单直观。实验数据管理功能帮助用户规范化地组织录制数据。支持快捷键操作，提高工作效率。

## 1.5 技术优势

### 1.5.1 分布式架构

系统采用Leader-Client分布式架构，Leader负责协调控制，Client负责实际录制。这种架构具有良好的可扩展性，可以根据需要增加或减少Client设备数量。

### 1.5.2 软件时钟同步

系统实现了简化的网络时间协议（SNTP），通过测量网络往返延迟，计算时钟偏移，实现Client设备与Leader设备的时钟同步。同步精度可达毫秒级。

### 1.5.3 自动服务发现

系统同时支持mDNS（多播DNS）和UDP广播两种服务发现机制。mDNS适用于支持Bonjour/Avahi的网络环境，UDP广播作为备选方案，确保在各种网络环境下都能正常工作。

### 1.5.4 可靠文件传输

文件传输采用TCP协议，确保可靠性。传输过程中采用分块传输，支持进度显示。传输完成后进行MD5校验，确保文件完整性。传输成功后自动删除Client端的本地文件，节省存储空间。

### 1.5.5 JavaFX图形界面

系统采用JavaFX框架开发图形界面，界面美观、响应流畅。支持实时更新客户端列表、文件传输进度等动态信息。

### 1.5.6 JavaCV视频处理

系统使用JavaCV（OpenCV的Java封装）进行视频录制和处理。JavaCV提供了丰富的视频处理功能，支持多种视频格式和编码器。

## 2.2 核心模块

### 2.2.1 同步控制模块

同步控制模块是系统的核心，负责Leader和Client之间的通信和协调。

**SoftwareSyncBase（基类）**
- 提供基础的RPC通信功能
- 管理UDP Socket的创建和监听
- 实现RPC消息的发送和接收
- 维护RPC回调函数映射表
- 实现时钟偏移管理

**SoftwareSyncLeader（Leader端）**
- 继承自SoftwareSyncBase
- 管理所有已连接的Client设备
- 处理Client的心跳消息
- 检测并移除过期的Client
- 广播RPC消息到所有Client
- 处理Client名称冲突检测
- 限制最大Client数量（10台）

**SoftwareSyncClient（Client端）**
- 继承自SoftwareSyncBase
- 定期发送心跳消息到Leader
- 接收Leader的控制指令
- 维护与Leader的连接状态
- 处理名称冲突和连接拒绝消息

### 2.2.2 服务发现模块

服务发现模块实现了Client自动发现Leader的功能，无需手动配置IP地址。

**LeaderDiscoveryService（Leader端）**
- 注册mDNS服务
- 定期发送UDP广播消息
- 广播消息包含Leader的IP地址和端口信息
- 广播消息包含认证令牌，防止误连接
- 支持多网卡环境，自动选择合适的网络接口

**ClientDiscoveryService（Client端）**
- 监听mDNS服务
- 监听UDP广播消息
- 解析Leader信息（IP、端口）
- 验证认证令牌
- 支持超时机制（5秒）
- 返回LeaderInfo对象，包含发现方式

### 2.2.3 文件传输模块

文件传输模块实现了Client到Leader的视频文件传输功能。

**FileReceiveServer（Leader端）**
- 监听TCP端口8246
- 接收Client的上传请求
- 验证文件是否已存在
- 接收文件数据块
- 计算并验证MD5校验和
- 按设备名称组织归档目录
- 提供文件接收事件回调

**FileUploadClient（Client端）**
- 连接到Leader的文件传输端口
- 发送上传请求（文件名、大小、MD5）
- 分块发送文件数据（64KB/块）
- 等待每个块的确认
- 接收验证结果
- 上传成功后删除本地文件

**FileTransferProtocol（传输协议）**
- 定义消息类型枚举
- 定义UploadRequest类（上传请求）
- 定义FileChunk类（文件数据块）
- 定义Response类（响应消息）
- 所有类实现Serializable接口，支持对象序列化

### 2.2.4 时间同步模块

时间同步模块实现了简化的网络时间协议（SNTP），用于同步Leader和Client的时钟。

**SimpleNetworkTimeProtocol**
- 监听UDP端口9428
- 处理时间同步请求
- 计算网络往返延迟
- 计算时钟偏移量
- 更新Client的时钟偏移
- 支持多轮同步，取最小延迟的结果

**时间同步流程**
1. Client发送时间戳T1到Leader
2. Leader接收时记录时间戳T2
3. Leader发送响应时记录时间戳T3
4. Client接收时记录时间戳T4
5. 计算往返延迟：RTT = (T4-T1) - (T3-T2)
6. 计算时钟偏移：Offset = ((T2-T1) + (T3-T4)) / 2
7. Client应用时钟偏移，实现与Leader的时钟同步

## 2.3 网络通信

### 2.3.1 RPC通信机制

系统采用基于UDP的RPC（远程过程调用）机制，实现Leader和Client之间的控制消息传递。

**RPC消息格式**
```
┌────────────┬──────────────────────┐
│ Method ID  │      Payload         │
│  (4 bytes) │    (variable)        │
└────────────┴──────────────────────┘
```

**Method ID分配**
- 0-999：系统保留
- 1000-199999：用户自定义
- 200000+：应用层RPC

**系统RPC方法**
- METHOD_HEARTBEAT (1)：心跳消息
- METHOD_HEARTBEAT_ACK (2)：心跳确认
- METHOD_OFFSET_UPDATE (3)：时钟偏移更新
- METHOD_MSG_NAME_CONFLICT (1106)：名称冲突通知
- METHOD_MSG_MAX_CLIENTS_REACHED (1107)：达到最大客户端数

**应用层RPC方法**
- METHOD_START_RECORDING (200003)：开始录制
- METHOD_STOP_RECORDING (200004)：停止录制
- METHOD_UPDATE_CLIENT_NAME (200005)：更新客户端名称

### 2.3.2 心跳机制

Client每秒向Leader发送一次心跳消息，Leader根据心跳消息更新Client的状态。

**心跳消息格式**
```
clientName,clientIP,synced
```

**心跳处理流程**
1. Client构造心跳消息，包含设备名称、IP地址、同步状态
2. Client通过UDP发送心跳消息到Leader的RPC端口
3. Leader接收心跳消息，解析设备信息
4. Leader检查是否是新Client，是否达到最大数量
5. Leader检查设备名称是否冲突
6. Leader更新Client信息，记录最后心跳时间
7. Leader发送心跳确认消息

**过期检测**
Leader每秒检查一次所有Client的最后心跳时间，如果超过2秒未收到心跳，则认为Client已断开连接，从列表中移除。

### 2.3.3 广播机制

Leader可以向所有已连接的Client广播RPC消息，实现统一控制。

**广播流程**
1. Leader调用broadcastRpc方法
2. 遍历所有已连接的Client
3. 向每个Client的RPC端口发送RPC消息
4. Client接收消息，调用对应的RPC回调函数

**典型应用**
- 开始录制：Leader广播START_RECORDING消息
- 停止录制：Leader广播STOP_RECORDING消息
- 更新视频参数：Leader广播参数更新消息

## 2.4 同步机制

### 2.4.1 时钟同步

系统采用软件时钟同步技术，通过测量网络延迟和计算时钟偏移，实现Client与Leader的时钟同步。

**同步精度**
- 理论精度：取决于网络延迟的稳定性
- 典型精度：1-10毫秒（局域网环境）
- 影响因素：网络负载、系统负载、网络拓扑

**同步维护**
- 初始同步：Client连接后立即进行时钟同步
- 定期同步：每小时重新同步一次
- 偏移应用：Client在所有时间戳上应用时钟偏移

### 2.4.2 录制同步

录制同步是指所有Client设备在同一时刻开始和停止录制。

**开始录制流程**
1. 用户在Leader端点击"开始录制"按钮
2. Leader生成录制参数（受试者ID、动作ID、回合ID等）
3. Leader广播START_RECORDING消息，包含录制参数
4. 所有Client接收消息，解析录制参数
5. Client创建视频文件，文件名包含录制参数
6. Client开始录制，记录开始时间戳
7. Client更新界面状态，显示"录制中"

**停止录制流程**
1. 用户在Leader端点击"停止录制"按钮
2. Leader广播STOP_RECORDING消息
3. 所有Client接收消息
4. Client停止录制，关闭视频文件
5. Client记录停止时间戳
6. Client更新界面状态，显示"录制完成"
7. Client自动开始上传视频文件到Leader

### 2.4.3 文件命名同步

所有Client使用统一的文件命名规范，确保文件名的一致性和可追溯性。

**文件名格式**
```
{设备名}_{受试者ID}_{动作ID}_{时间戳}_{回合ID}[_retake{N}].mp4
```

**示例**
- 正常录制：front_s01_m01_20240115143022_e1.mp4
- 重测录制：front_s01_m01_20240115143245_e1_retake1.mp4

**时间戳格式**
- yyyyMMddHHmmss
- 例如：20240115143022 表示 2024年1月15日 14:30:22

## 2.5 文件传输

### 2.5.1 传输协议

文件传输采用TCP协议，确保可靠性。传输过程分为三个阶段：请求、传输、验证。

**请求阶段**
1. Client连接到Leader的文件传输端口（8246）
2. Client发送UploadRequest对象，包含文件名、大小、MD5、设备名
3. Leader接收请求，检查文件是否已存在
4. Leader发送UPLOAD_ACCEPTED或UPLOAD_REJECTED响应

**传输阶段**
1. Client读取文件，分块发送（64KB/块）
2. 每个块包含块索引、数据、数据长度
3. Leader接收每个块，写入文件
4. Leader发送CHUNK_ACK确认
5. Client收到确认后发送下一块
6. 重复直到所有块发送完成

**验证阶段**
1. Leader计算接收文件的MD5校验和
2. Leader比较计算的MD5与请求中的MD5
3. 如果匹配，发送VERIFY_SUCCESS响应
4. 如果不匹配，删除文件，发送VERIFY_FAILED响应
5. Client收到成功响应后，删除本地文件

### 2.5.2 进度显示

系统提供实时的文件传输进度显示。

**Leader端显示**
- 当前接收的文件名和来源设备
- 传输进度百分比
- 已接收/总大小（MB）
- 进度条可视化

**Client端显示**
- 当前上传的文件名
- 传输进度百分比
- 已上传/总大小（MB）
- 进度条可视化

### 2.5.3 错误处理

系统对文件传输过程中可能出现的错误进行了完善的处理。

**常见错误**
- 文件不存在：Client端检查文件是否存在
- 文件已存在：Leader端拒绝上传
- 网络中断：TCP自动重连或超时失败
- MD5校验失败：Leader删除文件，Client保留本地文件
- 磁盘空间不足：Leader返回错误，Client保留本地文件

**错误恢复**
- 传输失败后，Client不删除本地文件
- 用户可以手动重试上传
- Leader端显示失败的文件列表

# 第三章 安装与配置

## 3.1 系统要求

### 3.1.1 硬件要求

**Leader端（控制端）**
- CPU：Intel Core i5或更高
- 内存：8GB或更高
- 硬盘：100GB可用空间（用于存储视频文件）
- 网卡：千兆以太网卡或WiFi 5/6
- 显示器：1920x1080或更高分辨率

**Client端（录制端）**
- CPU：Intel Core i5或更高
- 内存：4GB或更高
- 硬盘：20GB可用空间（临时存储）
- 网卡：千兆以太网卡或WiFi 5/6
- 摄像头：USB摄像头或内置摄像头
- 显示器：1280x720或更高分辨率

### 3.1.2 软件要求

**操作系统**
- Windows 10 64位或更高版本
- Windows 11 64位

**运行环境**
- Java Runtime Environment (JRE) 17或更高版本
- 系统已内置JRE，无需单独安装

**网络环境**
- 局域网连接（有线或无线）
- 所有设备在同一网段
- 网络带宽：建议100Mbps或更高
- 网络延迟：建议小于10ms

### 3.1.3 摄像头要求

**支持的摄像头类型**
- USB摄像头（推荐）
- 内置摄像头
- 网络摄像头（支持RTSP协议）

**摄像头参数**
- 分辨率：支持640x480、1280x720、1920x1080
- 帧率：支持15fps、24fps、30fps、60fps
- 接口：USB 2.0或更高

**推荐摄像头**
- 罗技C920/C922
- 微软LifeCam系列
- 其他支持UVC协议的USB摄像头

## 3.2 软件安装

### 3.2.1 获取软件

RecSync系统提供两种发布形式：

**便携版（推荐）**
- 文件名：RecSync-Portable-v1.0.zip
- 无需安装，解压即用
- 适合快速部署和移动使用

**安装版**
- 文件名：RecSync-Setup-v1.0.exe
- 标准Windows安装程序
- 自动创建开始菜单快捷方式
- 自动配置文件关联

### 3.2.2 安装步骤（便携版）

1. 下载RecSync-Portable-v1.0.zip文件
2. 解压到任意目录，例如：C:\RecSync
3. 解压后的目录结构：
```
RecSync/
├── RecSync-Leader.exe      # Leader端启动程序
├── RecSync-Client.exe      # Client端启动程序
├── lib/                    # 依赖库目录
├── jre/                    # Java运行环境
├── config/                 # 配置文件目录
└── README.txt              # 说明文件
```

4. 双击RecSync-Leader.exe启动Leader端
5. 双击RecSync-Client.exe启动Client端

### 3.2.3 安装步骤（安装版）

1. 下载RecSync-Setup-v1.0.exe文件
2. 双击运行安装程序
3. 选择安装语言（简体中文）
4. 阅读并接受许可协议
5. 选择安装目录（默认：C:\Program Files\RecSync）
6. 选择开始菜单文件夹
7. 选择是否创建桌面快捷方式
8. 点击"安装"开始安装
9. 安装完成后，点击"完成"

### 3.2.4 首次运行

**Leader端首次运行**
1. 启动RecSync-Leader.exe
2. 系统自动初始化服务
3. 显示Leader的IP地址和端口
4. 等待Client连接

**Client端首次运行**
1. 启动RecSync-Client.exe
2. 系统自动搜索Leader
3. 如果找到Leader，自动连接
4. 如果未找到，提示手动输入Leader IP地址
5. 连接成功后，显示连接状态
